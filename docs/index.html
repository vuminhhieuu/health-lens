BMAD v6 & IDE

BMAD (Build More Architect Dreams) là AI-driven development framework. Hiểu BMAD là hiểu cách team tổ chức công việc từ ý tưởng đến code.

💡BMAD là source of truth. Mọi story bắt nguồn từ BMAD artifacts. Linear là downstream — issue US-XX trên Linear được tạo từ story file BMAD, không phải ngược lại.

BMAD là gì

BMAD là framework giúp AI agents làm việc nhất quán xuyên suốt vòng đời phần mềm. Thay vì hỏi AI chung chung, BMAD tổ chức context thành 4 phases — mỗi phase tạo ra artifacts làm input cho phase tiếp theo.

Ai làm gì
Team leadOpencode — chạy Phase 1–3, tạo story
DevCursor/VS Code — implement story với bmad-dev
Agents chính
bmad-smScrum Master — tạo story, sprint planning
bmad-devDeveloper — implement + code review
bmad-pmPM — PRD, epics (đã dùng)
bmad-architectArchitect — architecture.md (đã dùng)

Cấu trúc thư mục BMAD

_bmad/
  bmm/config.yaml              # BMAD config của project

_bmad-output/
  prd.md                       # Product Requirements Document
  architecture.md              # Technical architecture decisions
  project-context.md           # "Constitution" cho AI agents ← QUAN TRỌNG
  planning-artifacts/
    epics.md                   # Epic breakdown
  implementation-artifacts/
    sprint-status.yaml         # Track progress sprint hiện tại
    stories/
      story-US-01-ten.md       # Story file — dev entry point
      story-US-02-ten.md
      ...

project-context.md — File quan trọng nhất

Đây là "constitution" của project — mọi implementation workflow tự động load file này. Không có nó, AI của mỗi dev đưa ra quyết định khác nhau → code inconsistent.

⚠File này phải được tạo trước Sprint 1. Chạy bmad-generate-project-context từ architecture.md đang có, sau đó review và bổ sung.

Phần 1: Tech stack & versions

## Technology Stack & Versions
- Java 21, Spring Boot 3.x
- Next.js 14 (App Router), TypeScript 5.x
- React Native 0.73, Expo SDK 50
- PostgreSQL 16
- Testing: JUnit 5, Jest, Detox

Phần 2: Critical rules

## Critical Implementation Rules
- API calls dùng apiClient — không fetch trực tiếp
- Dữ liệu y tế: mã hóa trước khi lưu DB
- Error handling: dùng GlobalExceptionHandler
- TypeScript: strict mode, không dùng any
- Components: co-located test files

Workflow dev hàng ngày với BMAD

📖Story file là self-contained — chứa đủ what, why, how, acceptance criteria, và tasks. Dev không cần đọc PRD hay architecture doc trực tiếp.

Khi nhận story US-01

  1. Linear: Tìm issue US-01 → Ctrl+Shift+. copy branch → issue tự chuyển In Progress
  2. Git: git checkout -b feat/US-01-ten-tinh-nang
  3. IDE: Mở _bmad-output/implementation-artifacts/stories/story-US-01-ten.md → đọc ACs và tasks
  4. BMAD: Chạy bmad-dev → trigger DS (Dev Story) → AI implement theo story + project-context.md
  5. BMAD: Chạy bmad-dev → trigger CR (Code Review) → AI self-review trước khi push
  6. Verify: Kiểm tra sprint-status.yaml đã cập nhật đúng (auto nhưng verify lại)
  7. GitHub: Tạo PR với Fixes US-01 trong body description

Convention tên story file

Opencode — Setup BMAD (Team Lead)

  1. Opencode đã được team lead cài sẵn với BMAD skills
  2. Workflow team lead: bmad-sm → CS (Create Story) → tạo story-US-XX file

💡Opencode là CLI tool — phù hợp cho team lead vì có thể chạy multi-step workflows không cần IDE UI. Dev dùng Cursor/VS Code cho implementation vẫn hoàn toàn ổn.

Hướng dẫn công cụ

Tất cả công cụ team dùng — cách hoạt động, cách dùng, app tương tự, và tài liệu tham khảo.

Workflows

Tất cả quy trình team thực hiện — từ bắt đầu ngày làm việc đến sprint planning, bug handling, và hotfix.

Bắt đầu ngày làm việc

💡Toàn bộ standup là async — không có meeting bắt buộc. Reply Geekbot DM bất kỳ lúc nào trước 9:00.
  1. 8:00 — Đọc DM Google Calendar: hôm nay có meeting gì, cần chuẩn bị gì
  2. 8:00 — Reply DM Geekbot với 3 câu hỏi standup (trả lời trước 9:00)
    Câu 1: hôm qua làm gì · Câu 2: hôm nay sẽ làm gì · Câu 3: có blocker không
  3. 9:00 — Đọc #standup: ai đang làm gì, ai có blocker cần support
  4. 9:00+ — Check #dev-mr: có PR nào được assign cho mình không (rule: review trong 24h)
  5. Mở Linear → Cycle hiện tại → chọn task tiếp theo theo priority

Cuối ngày (tùy chọn)

  • Push code đang làm dù chưa xong — tránh mất code, team biết progress
  • Update issue status nếu cần (ví dụ: blocked → comment lý do vào issue)
  • Nếu PR đang chờ review → nhắc reviewer trong thread nếu quá 24h

Làm 1 task từ đầu đến cuối

⚠Bước 1 phải làm trước: add issue vào Cycle rồi mới bắt đầu code. Issue không có trong Cycle sẽ không tính vào sprint metrics.
  1. Nhận task từ Cycle
    Linear → Cycles → Sprint hiện tại → chọn issue Todo → assign cho mình nếu chưa assign
  2. Đảm bảo issue đã có trong Cycle
    Nếu chưa: mở issue → Cycle → chọn sprint hiện tại. Bắt buộc trước khi bắt đầu code.
  3. Copy branch name từ Linear và checkout
    Linear: Ctrl+Shift+. → issue tự chuyển In Progress ngay lập tức 
    git checkout main
git pull origin main          # luôn pull main mới nhất trước
git checkout -b feat/LIN-42-ten-issue
  4. Code và commit theo quy ước
    git add .
git commit -m "feat(LIN-42): add user login with JWT"
git push origin feat/LIN-42-ten-issue
    Commit thường xuyên (không phải chờ xong mới commit) — dễ review, dễ revert
  5. Tạo PR trên GitHub
    Title: feat(LIN-42): ten tinh nang · Description: điền theo template, bắt buộc có Fixes LIN-42
    Assign 1 reviewer · Đợi CI pass trước khi request review
    → Issue tự chuyển In Review · Reviewer nhận DM từ Sprint Bot
  6. Xử lý review comments
    Nếu có Request changes: fix → commit mới → push → reply comment "Done" để reviewer biết
  7. Merge sau khi approved + CI pass
    Squash and merge → issue tự chuyển Done → Apps Script ghi mr_stats → tác giả nhận DM xác nhận

Xử lý conflict với main

⚠Nếu branch bị conflict với main: author tự rebase, không để reviewer làm.
git checkout main
git pull origin main
git checkout feat/LIN-42-ten
git rebase main               # resolve conflicts nếu có
git push origin feat/LIN-42-ten --force-with-lease

Review Pull Request

⏱Rule: respond trong 24 giờ sau khi được assign reviewer. Nếu không thể review đúng hạn → comment vào PR để author biết.
  1. Nhận DM từ Sprint Bot
    DM có link PR và tên author. Click vào link hoặc tìm trong #dev-mr.
  2. Đọc context trước khi xem code
    Mở issue Linear tương ứng → hiểu requirement · Đọc PR description: What / Why / How
  3. Review code (tab Files changed)
    Comment inline vào đúng dòng · Dùng "nit:" prefix cho comment không blocking
  4. Quyết định
    Trường hợpAction
    Code ổn, không có issueApprove → bấm nút Approve, không chỉ comment "LGTM"
    Có vấn đề nhỏ (style, naming)Approve + comment với "nit:" prefix
    Có vấn đề cần fix trước khi mergeRequest changes + comment rõ cần sửa gì
    Không hiểu code, cần hỏiComment câu hỏi, không Request changes chỉ vì không hiểu
  5. Merge (nếu là reviewer được assign merge)
    Sau khi approve + CI pass → Squash and merge → Confirm merge

Sau khi reviewer Request changes

  1. Author fix theo comments → commit → push
  2. Author reply "Done, đã fix ở commit abc123" vào từng comment
  3. Author re-request review (nút trên GitHub)
  4. Reviewer kiểm tra lại → Approve nếu ổn

Bug report & fix

🤖Issue có label bug được Apps Script tự động track trong Google Sheets tab bug_log — ghi ngày tạo và ngày resolve để tính resolution_hours.

Khi phát hiện bug

  1. Tạo issue trong Linear
    Title rõ ràng: "[Bug] mô tả bug" · Label: bug (bắt buộc lowercase) · Priority: Urgent/High/Medium tùy severity
    Description: Steps to reproduce, Expected vs Actual behavior, Environment
    → Apps Script tự ghi vào bug_log với date_reported ngay khi issue tạo
  2. Assign cho người fix
    Add vào Cycle hiện tại nếu cần fix trong sprint này. Bug severity cao → Urgent priority → ưu tiên hơn task thường.
  3. Fix theo workflow task bình thường
    Branch: fix/LIN-XX-mo-ta-bug · Commit: fix(LIN-XX): ...
  4. Tạo PR với Fixes LIN-XX → merge
    Khi issue chuyển sang Done → Apps Script cập nhật bug_log với date_resolved và tính resolution_hours.

Bug severity guide

SeverityMô tảPriority LinearTarget fix
CriticalProduction down, data loss, security breachUrgent → HotfixNgay lập tức
HighFeature chính không hoạt động, ảnh hưởng nhiều userHighTrong sprint hiện tại
MediumFeature phụ lỗi, có workaroundMediumSprint tiếp theo
LowUI glitch, typo, ảnh hưởng nhỏLowBacklog

Hotfix — Production bug khẩn cấp

🚨Hotfix chỉ dùng cho bug Critical trên production cần fix ngay, không thể chờ đến sprint tiếp theo.
  1. Tạo issue Linear với priority Urgent
    Label: bug · Priority: Urgent · Tag "hotfix" vào title
  2. Tạo hotfix branch từ main
    git checkout main
git pull origin main
git checkout -b hotfix/LIN-99-payment-crash
  3. Fix và test tối thiểu
    Focus vào đúng bug, không refactor hay add feature thêm
  4. Tạo PR → review nhanh (target: <2h)
    Notify reviewer trực tiếp trên Slack, không chờ DM từ bot · Reviewer ưu tiên review hotfix hơn task thường
  5. Merge → Deploy ngay
    Sau merge: monitor logs ít nhất 30 phút để đảm bảo fix không gây side effect
  6. Post-mortem (sau khi ổn định)
    Comment vào issue: root cause, fix approach, cách prevent lần sau

Sprint Planning & Review

Đầu sprint — Sprint Planning

  1. Team lead: chuẩn bị trước meeting
    Tạo Cycle mới trong Linear với ngày bắt đầu và kết thúc · Update CURRENT_SPRINT trong Code.gs → deploy
  2. Review backlog cùng team (15–30 phút)
    Kéo issues từ Backlog vào Cycle mới · Thống nhất priority · Clarify requirement nếu cần
  3. Estimate story points
    Mỗi issue gán số story points (hoặc T-shirt size) · Tổng không vượt quá team capacity
    PointsT-shirtÝ nghĩa
    1XSDưới 2h, rõ ràng hoàn toàn
    2SNửa ngày, ít uncertainty
    3M1 ngày, hiểu rõ
    5L2–3 ngày, có một số ẩn
    8XLNên tách nhỏ hơn
  4. Assign issues
    Mỗi người nhận issues theo capacity và expertise · Không assign quá nhiều cho 1 người

Cuối sprint — Sprint Review & Retro

  1. Team lead gõ /sprint-summary trong #sprint
    Apps Script pull data từ Linear + GitHub → post summary với metrics và breakdown theo người
  2. Review metrics cùng team
    Completion rate · Carry-over count · Bug count · Avg PR review time · Ai bị blocker nhiều
  3. Sprint Retrospective (15–20 phút)
    3 câu hỏi: (1) Gì tốt cần giữ? (2) Gì cần cải thiện? (3) Blockers nào lặp lại?
  4. Xử lý carry-over issues
    Issues chưa done → quyết định: add vào sprint mới, deprioritize, hoặc break down nhỏ hơn
  5. Chuẩn bị backlog cho sprint tiếp theo
    Tạo Cycle mới · Rough priority backlog cho buổi planning tiếp

OOO — Nghỉ phép

  1. Thông báo trước (ít nhất 1 ngày)
    Nhắn vào #general hoặc DM team lead · Update Google Calendar
  2. Set OOO trong Geekbot
    OOO     ← nhắn vào DM Geekbot bot → chọn ngày bắt đầu và kết thúc
    Geekbot sẽ skip DM standup và ghi "Out-of-office" trong tổng hợp #standup những ngày đó
  3. Handoff nếu cần
    Comment vào các issues đang làm: trạng thái hiện tại, blocker nếu có, ai có thể tiếp nếu urgent
  4. Khi quay lại
    back    ← nhắn vào DM Geekbot bot → quay lại standup bình thường
💡Nghỉ đột xuất: nhắn OOO vào Geekbot trước 8:00 sáng ngày đó là đủ. Không cần approval.
